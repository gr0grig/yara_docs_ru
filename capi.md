# C API

Вы можете интегрировать YARA в свой проект на C/C++, используя API, предоставляемый библиотекой *libyara*. Этот API предоставляет доступ ко всем возможностям YARA и является тем же API, который используется утилитами командной строки `yara` и `yarac`.

## Инициализация и завершение работы *libyara*

Первое, что должна сделать ваша программа при использовании *libyara* -- это инициализировать библиотеку. Это делается вызовом функции `yr_initialize`. Эта функция выделяет все ресурсы, необходимые библиотеке, и инициализирует внутренние структуры данных. Её противоположностью является `yr_finalize`, которая должна быть вызвана, когда вы закончите использование библиотеки.

В многопоточной программе только главный поток должен вызывать `yr_initialize` и `yr_finalize`. Никаких дополнительных действий от других потоков, использующих библиотеку, не требуется.

## Компиляция правил

Прежде чем использовать ваши правила для сканирования каких-либо данных, необходимо скомпилировать их в бинарную форму. Для этого вам понадобится компилятор YARA, который можно создать с помощью `yr_compiler_create`. После использования компилятор должен быть уничтожен с помощью `yr_compiler_destroy`.

Вы можете использовать `yr_compiler_add_file`, `yr_compiler_add_fd` или `yr_compiler_add_string` для добавления одного или нескольких источников для компиляции. Обе эти функции принимают необязательное пространство имён. Правила, добавленные в одно и то же пространство имён, ведут себя так, как если бы они содержались в одном исходном файле или строке, поэтому идентификаторы правил должны быть уникальными среди всех источников, разделяющих пространство имён. Если аргумент namespace равен `NULL`, правила помещаются в пространство имён *по умолчанию*.

Функции `yr_compiler_add_file`, `yr_compiler_add_fd` и `yr_compiler_add_string` возвращают количество ошибок, найденных в исходном коде. Если правила корректны, они вернут 0. Если любая из этих функций вернёт ошибку, компилятор больше нельзя использовать ни для добавления новых правил, ни для получения скомпилированных правил.

Для получения подробной информации об ошибках необходимо установить функцию обратного вызова с помощью `yr_compiler_set_callback` перед вызовом любой из функций компиляции. Функция обратного вызова имеет следующий прототип:

```c
void callback_function(
    int error_level,
    const char* file_name,
    int line_number,
    const YR_RULE* rule,
    const char* message,
    void* user_data)
```

> Изменено в версии 4.0.0

Возможные значения для `error_level` -- это `YARA_ERROR_LEVEL_ERROR` и `YARA_ERROR_LEVEL_WARNING`. Аргументы `file_name` и `line_number` содержат имя файла и номер строки, где произошла ошибка или предупреждение. `file_name` -- это то имя, которое было передано в `yr_compiler_add_file` или `yr_compiler_add_fd`. Оно может быть `NULL`, если вы передали `NULL` или если вы используете `yr_compiler_add_string`. `rule` -- это указатель на структуру `YR_RULE`, представляющую правило, которое содержит ошибку, но он может быть `NULL`, если ошибка не содержится в конкретном правиле. Указатель `user_data` -- тот же, который вы передали в `yr_compiler_set_callback`.

По умолчанию для правил, содержащих ссылки на другие файлы (`include "filename.yara"`), YARA будет пытаться найти эти файлы на диске. Однако, если вы хотите получать импортируемые правила из другого источника (например, из базы данных или удалённого сервиса), можно установить функцию обратного вызова с помощью `yr_compiler_set_include_callback`.

Этот обратный вызов получает следующие параметры:

* `include_name`: имя запрашиваемого файла.
* `calling_rule_filename`: имя файла, из которого выполняется запрос (NULL, если это не файл).
* `calling_rule_namespace`: пространство имён (NULL, если не определено).
* `user_data`: тот же указатель, переданный в `yr_compiler_set_include_callback`.

Он должен возвращать содержимое запрашиваемого файла в виде строки с нулевым завершением. Память для этой строки должна быть выделена функцией обратного вызова. Как только станет безопасно освободить память, использованную для возврата результата обратного вызова, будет вызвана функция include_free, переданная в `yr_compiler_set_include_callback`. Если память не нужно освобождать, вместо include_free можно передать NULL. Вы можете полностью отключить поддержку включений, установив NULL в качестве функции обратного вызова с помощью `yr_compiler_set_include_callback`.

Функция обратного вызова имеет следующий прототип:

```c
const char* include_callback(
    const char* include_name,
    const char* calling_rule_filename,
    const char* calling_rule_namespace,
    void* user_data);
```

Функция освобождения имеет следующий прототип:

```c
void include_free(
    const char* callback_result_ptr,
    void* user_data);
```

После успешного добавления некоторых источников вы можете получить скомпилированные правила с помощью функции `yr_compiler_get_rules`. Вы получите указатель на структуру `YR_RULES`, которая может быть использована для сканирования ваших данных, как описано в разделе [Сканирование данных](#сканирование-данных). После вызова `yr_compiler_get_rules` вы больше не можете добавлять источники в компилятор, но можете вызывать `yr_compiler_get_rules` несколько раз. Каждый раз при вызове этой функции она возвращает указатель на ту же структуру `YR_RULES`. Обратите внимание, что это поведение является новым в YARA 4.0.0; в YARA 3.X и 2.X `yr_compiler_get_rules` возвращала новую копию структуры `YR_RULES`.

Экземпляры `YR_RULES` должны быть уничтожены с помощью `yr_rules_destroy`.

## Определение внешних переменных

Если ваши правила используют внешние переменные (как в примере ниже), вы должны определить эти переменные с помощью одной из функций `yr_compiler_define_XXXX_variable`. Переменные должны быть определены до компиляции правил с помощью `yr_compiler_add_XXXX`, и они должны быть определены с типом, соответствующим контексту, в котором переменная используется в правиле. Переменная, которая используется как `my_var == 5`, не может быть определена как строковая переменная.

При определении внешних переменных с помощью `yr_compiler_define_XXXX_variable` вы должны предоставить значение для каждой переменной. Это значение встраивается в скомпилированные правила и используется всякий раз, когда переменная появляется в правиле. Однако вы можете изменить значение, связанное с внешней переменной, после компиляции правил, используя любую из функций `yr_rules_define_XXXX_variable`.

## Сохранение и загрузка скомпилированных правил

Скомпилированные правила могут быть сохранены в файл и загружены позже с помощью `yr_rules_save` и `yr_rules_load`. Правила, скомпилированные и сохранённые на одной машине, могут быть загружены на другой машине, при условии одинакового порядка байтов (endianness), независимо от операционной системы или разрядности системы (32-бит или 64-бит). Однако файлы, сохранённые более ранними версиями YARA, могут не работать с новыми версиями из-за изменений в формате файла.

Вы также можете сохранять и загружать ваши правила из обобщённых потоков данных, используя функции `yr_rules_save_stream` и `yr_rules_load_stream`. Эти функции получают указатель на структуру `YR_STREAM`, определённую следующим образом:

```c
typedef struct _YR_STREAM
{
  void* user_data;

  YR_STREAM_READ_FUNC read;
  YR_STREAM_WRITE_FUNC write;

} YR_STREAM;
```

Вы должны предоставить свою собственную реализацию функций `read` и `write`. Функция `read` используется функцией `yr_rules_load_stream` для чтения данных из вашего потока, а функция `write` используется функцией `yr_rules_save_stream` для записи данных в ваш поток.

Ваши функции `read` и `write` должны соответствовать следующим прототипам:

```c
size_t read(
    void* ptr,
    size_t size,
    size_t count,
    void* user_data);

size_t write(
    const void* ptr,
    size_t size,
    size_t count,
    void* user_data);
```

Аргумент `ptr` -- это указатель на буфер, в который функция `read` должна поместить прочитанные данные, или в котором функция `write` найдёт данные, которые необходимо записать в поток. В обоих случаях `size` -- это размер каждого элемента, читаемого или записываемого, а `count` -- количество элементов. Общий размер читаемых или записываемых данных равен `size` * `count`. Функция `read` должна возвращать количество прочитанных элементов, функция `write` должна возвращать общее количество записанных элементов.

Указатель `user_data` -- тот же, который вы указали в структуре `YR_STREAM`. Вы можете использовать его для передачи произвольных данных в ваши функции `read` и `write`.

## Сканирование данных

После получения экземпляра `YR_RULES` вы можете использовать его напрямую с одной из описанных ниже функций `yr_rules_scan_XXXX` или создать сканер с помощью `yr_scanner_create`. Давайте начнём с обсуждения первого подхода.

Структура `YR_RULES`, полученная от компилятора, может быть использована с `yr_rules_scan_file`, `yr_rules_scan_fd` или `yr_rules_scan_mem` для сканирования файла, файлового дескриптора и буфера в памяти соответственно. Результаты сканирования возвращаются в вашу программу через функцию обратного вызова. Обратный вызов имеет следующий прототип:

```c
int callback_function(
    YR_SCAN_CONTEXT* context,
    int message,
    void* message_data,
    void* user_data);
```

Возможные значения для `message`:

```
CALLBACK_MSG_RULE_MATCHING
CALLBACK_MSG_RULE_NOT_MATCHING
CALLBACK_MSG_SCAN_FINISHED
CALLBACK_MSG_IMPORT_MODULE
CALLBACK_MSG_MODULE_IMPORTED
CALLBACK_MSG_TOO_MANY_MATCHES
CALLBACK_MSG_CONSOLE_LOG
```

Ваша функция обратного вызова будет вызвана один раз для каждого правила с сообщением либо `CALLBACK_MSG_RULE_MATCHING`, либо `CALLBACK_MSG_RULE_NOT_MATCHING`, в зависимости от того, совпадает правило или нет. В обоих случаях указатель на структуру `YR_RULE`, связанную с правилом, передаётся в аргументе `message_data`. Вам нужно лишь выполнить приведение типа из `void*` к `YR_RULE*`, чтобы получить доступ к структуре. Вы можете управлять тем, вызывает ли YARA вашу функцию обратного вызова с сообщениями `CALLBACK_MSG_RULE_MATCHING` и `CALLBACK_MSG_RULE_NOT_MATCHING`, используя флаги `SCAN_FLAGS_REPORT_RULES_MATCHING` и `SCAN_FLAGS_REPORT_RULES_NOT_MATCHING`, как описано далее в этом разделе.

Этот обратный вызов также вызывается с сообщением `CALLBACK_MSG_IMPORT_MODULE`. Все модули, на которые ссылается оператор `import` в правилах, импортируются один раз для каждого сканируемого файла. В этом случае `message_data` указывает на структуру `YR_MODULE_IMPORT`. Эта структура содержит поле `module_name`, указывающее на строку с нулевым завершением с именем импортируемого модуля, и два других поля `module_data` и `module_data_size`. Эти поля изначально установлены в `NULL` и `0`, но ваша программа может присвоить указатель на некоторые произвольные данные полю `module_data`, одновременно установив `module_data_size` в размер данных. Таким образом вы можете передавать дополнительные данные модулям, которые их требуют, например, модулю Cuckoo.

После импорта модуля обратный вызов вызывается снова с сообщением `CALLBACK_MSG_MODULE_IMPORTED`. В этом случае `message_data` указывает на структуру `YR_OBJECT_STRUCTURE`. Эта структура содержит всю информацию, предоставленную модулем о текущем сканируемом файле.

Если во время сканирования строка достигает максимального количества совпадений, ваш обратный вызов будет вызван один раз с сообщением `CALLBACK_MSG_TOO_MANY_MATCHES`. В этом случае `message_data` -- это `YR_STRING*`, который указывает на строку, вызвавшую предупреждение. Если ваш обратный вызов возвращает `CALLBACK_CONTINUE`, строка будет отключена и сканирование продолжится, в противном случае сканирование будет остановлено.

Ваш обратный вызов будет вызван из модуля console с сообщением `CALLBACK_MSG_CONSOLE_LOG`. В этом случае аргумент `message_data` будет `char*`, который является строкой, сгенерированной модулем console. Ваш обратный вызов может делать с этой строкой всё что угодно, включая логирование в внешний источник логов или вывод в stdout.

Наконец, функция обратного вызова также вызывается с сообщением `CALLBACK_MSG_SCAN_FINISHED` по завершении сканирования. В этом случае `message_data` равен `NULL`.

Обратите внимание, что вы не должны вызывать какие-либо из функций `yr_rules_scan_XXXX` из функции обратного вызова, так как эти функции не являются реентерабельными.

Ваша функция обратного вызова должна возвращать одно из следующих значений:

```
CALLBACK_CONTINUE
CALLBACK_ABORT
CALLBACK_ERROR
```

Если она возвращает `CALLBACK_CONTINUE`, YARA продолжит нормальную работу. `CALLBACK_ABORT` прервёт сканирование, но результатом функции `yr_rules_scan_XXXX` будет `ERROR_SUCCESS`. С другой стороны, `CALLBACK_ERROR` также прервёт сканирование, но результатом `yr_rules_scan_XXXX` будет `ERROR_CALLBACK_ERROR`.

Аргумент `user_data`, переданный в вашу функцию обратного вызова, -- тот же, который вы передали в `yr_rules_scan_XXXX`. Этот указатель не затрагивается YARA, он является лишь способом передачи произвольных данных в функцию обратного вызова.

Все функции `yr_rules_scan_XXXX` принимают аргумент `flags`, который позволяет настраивать некоторые аспекты процесса сканирования. Поддерживаемые флаги:

* `SCAN_FLAGS_FAST_MODE`
* `SCAN_FLAGS_NO_TRYCATCH`
* `SCAN_FLAGS_REPORT_RULES_MATCHING`
* `SCAN_FLAGS_REPORT_RULES_NOT_MATCHING`

Флаг `SCAN_FLAGS_FAST_MODE` немного ускоряет сканирование, избегая множественных совпадений одной и той же строки, когда это не необходимо. После нахождения строки в файле она впоследствии игнорируется, что означает, что у вас будет одно совпадение для строки, даже если она появляется несколько раз в сканируемых данных. Этот флаг имеет тот же эффект, что и параметр командной строки `-f`.

`SCAN_FLAGS_REPORT_RULES_MATCHING` и `SCAN_FLAGS_REPORT_RULES_NOT_MATCHING` управляют тем, вызывается ли обратный вызов для правил, которые совпадают, или для правил, которые не совпадают, соответственно. Если указан только `SCAN_FLAGS_REPORT_RULES_MATCHING`, обратный вызов будет вызван для совпавших правил с сообщением `CALLBACK_MSG_RULE_MATCHING`, но не будет вызван для несовпавших правил. Если указан только `SCAN_FLAGS_REPORT_RULES_NOT_MATCHING`, произойдёт обратное -- обратный вызов будет вызван с сообщениями `CALLBACK_MSG_RULE_NOT_MATCHING`, но не с сообщениями `CALLBACK_MSG_RULE_MATCHING`. Если оба флага скомбинированы вместе (по умолчанию), обратный вызов будет вызван как для совпавших, так и для несовпавших правил. Для обратной совместимости, если ни один из этих двух флагов не указан, сканер будет следовать поведению по умолчанию.

Кроме того, функции `yr_rules_scan_XXXX` могут принимать аргумент `timeout`, который принудительно прерывает сканирование после указанного количества секунд (приблизительно). Если `timeout` равен 0, это означает отсутствие тайм-аута.

### Использование сканера

Функций `yr_rules_scan_XXXX` достаточно в большинстве случаев, но иногда вам может потребоваться более детальный контроль над сканированием. В таких случаях вы можете создать сканер с помощью `yr_scanner_create`. Сканер -- это просто обёртка вокруг структуры `YR_RULES`, которая хранит дополнительную конфигурацию, такую как внешние переменные, не влияя на других пользователей структуры `YR_RULES`.

Сканер особенно полезен, когда вы хотите использовать одну и ту же структуру `YR_RULES` с несколькими рабочими процессами (это может быть отдельный поток, сопрограмма и т.д.), и каждому рабочему процессу необходимо задавать различные значения внешних переменных. В этом случае нельзя использовать `yr_rules_define_XXXX_variable` для установки значений внешних переменных, так как каждый рабочий процесс, использующий `YR_RULES`, будет затронут такими изменениями. Однако каждый рабочий процесс может иметь собственный сканер, где сканеры разделяют одну и ту же `YR_RULES`, и использовать `yr_scanner_define_XXXX_variable` для установки внешних переменных, не влияя на остальные рабочие процессы.

Это лучшее решение, чем иметь отдельную `YR_RULES` для каждого рабочего процесса, поскольку структуры `YR_RULES` имеют большой объём памяти (особенно при большом количестве правил), тогда как сканеры очень легковесны.

## Справочник по API

### Структуры данных

#### YR_COMPILER

Структура данных, представляющая компилятор YARA.

#### YR_SCAN_CONTEXT

Структура данных, содержащая информацию о текущем сканировании. Указатель на эту структуру передаётся в функцию обратного вызова, которая получает уведомления о найденных совпадениях. Эта структура также используется для итерации по результатам.

#### YR_MATCH

Структура данных, представляющая совпадение строки.

* `int64_t base`

  Базовое смещение/адрес совпадения. При сканировании файла это поле обычно равно нулю, при сканировании памяти процесса это поле является виртуальным адресом блока памяти, где было найдено совпадение.

* `int64_t offset`

  Смещение совпадения относительно *base*.

* `int32_t match_length`

  Длина совпавшей строки.

* `const uint8_t* data`

  Указатель на буфер, содержащий часть совпавшей строки.

* `int32_t data_length`

  Длина буфера `data`. `data_length` -- это минимум из `match_length` и `MAX_MATCH_DATA`.

> Изменено в версии 3.5.0

#### YR_META

Структура данных, представляющая значение метаданных.

* `const char* identifier`

  Идентификатор метаданных.

* `int32_t type`

  Один из следующих типов метаданных:

  * `META_TYPE_INTEGER`
  * `META_TYPE_STRING`
  * `META_TYPE_BOOLEAN`

#### YR_MODULE_IMPORT

* `const char* module_name`

  Имя импортируемого модуля.

* `void* module_data`

  Размер дополнительных данных, передаваемых модулю. Ваша программа должна установить соответствующее значение, если `module_data` изменяется.

#### YR_RULE

Структура данных, представляющая одно правило.

* `const char* identifier`

* `const char* tags`

* `YR_STRING* strings`

  Указатель на последовательность структур `YR_STRING`. Для итерации по структурам используйте `yr_rule_strings_foreach`.

* `YR_NAMESPACE* ns`

  Указатель на структуру `YR_NAMESPACE`.

Структура данных, представляющая набор скомпилированных правил.

#### YR_STREAM

#### `int yr_scanner_scan_mem_blocks(YR_SCANNER* scanner, YR_MEMORY_BLOCK_ITERATOR* iterator)`

> Добавлено в версии 3.8.0

Сканирование серии блоков памяти, предоставляемых через `YR_MEMORY_BLOCK_ITERATOR`. Итератор имеет пару функций `first` и `next`, которые должны возвращать первый и следующий блоки соответственно. Когда эти функции возвращают `NULL`, это означает, что больше нет блоков для сканирования.

В YARA 4.1 и более поздних версиях функции `first` и `next` могут возвращать `NULL` и устанавливать поле `last_error` в `YR_MEMORY_BLOCK_ITERATOR` в значение `ERROR_BLOCK_NOT_READY`. Это означает, что итератор пока не может вернуть следующий блок, но операция может быть повторена. В таких случаях `yr_scanner_scan_mem_blocks` также возвращает `ERROR_BLOCK_NOT_READY`, но сканер сохраняет своё состояние, и эта функция может быть вызвана снова для продолжения сканирования с того места, где оно было приостановлено. Это может быть сделано несколько раз, пока блок не будет готов и итератор сможет его вернуть.

Обратите внимание, однако, что после того, как итератор завершит полную итерацию, любая последующая итерация должна проходить без возврата `ERROR_BLOCK_NOT_READY`. Во время первой итерации итератор должен сохранять в памяти всю необходимую информацию о блоках, чтобы повторная итерация могла быть выполнена без обращения к дорогостоящим операциям, которые могут привести к ошибке `ERROR_BLOCK_NOT_READY`.

Возвращает один из следующих кодов ошибок:

* `ERROR_SUCCESS`
* `ERROR_INSUFFICIENT_MEMORY`
* `ERROR_SCAN_TIMEOUT`
* `ERROR_CALLBACK_ERROR`
* `ERROR_TOO_MANY_MATCHES`
* `ERROR_BLOCK_NOT_READY`

Файл правил повреждён.

#### `ERROR_UNSUPPORTED_FILE_VERSION`

Файл был создан другой версией YARA и не может быть загружен текущей версией.

#### `ERROR_SCAN_TIMEOUT`

Превышено время ожидания сканирования.

#### `ERROR_CALLBACK_ERROR`

Обратный вызов вернул ошибку.

#### `ERROR_TOO_MANY_MATCHES`

Слишком много совпадений для некоторой строки в ваших правилах. Это обычно происходит, когда ваши правила содержат очень короткие или очень распространённые строки, такие как `01 02` или `FF FF FF FF`. Предел определяется значением `YR_MAX_STRING_MATCHES` в файле *./include/yara/limits.h*.

#### `ERROR_BLOCK_NOT_READY`

Следующий блок памяти для сканирования не готов; пользовательские итераторы могут возвращать этот код.
