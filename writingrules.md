# Написание правил YARA

Правила YARA легко писать и понимать, их синтаксис напоминает язык C. Вот простейшее правило, которое можно написать для YARA и которое абсолютно ничего не делает:

```yara
rule dummy
{
    condition:
        false
}
```

Каждое правило в YARA начинается с ключевого слова `rule`, за которым следует идентификатор правила. Идентификаторы должны соответствовать тем же лексическим соглашениям, что и в языке программирования C: они могут содержать любые буквенно-цифровые символы и символ подчёркивания, но первый символ не может быть цифрой. Идентификаторы правил чувствительны к регистру и не могут превышать 128 символов. Следующие ключевые слова являются зарезервированными и не могут использоваться в качестве идентификаторов:

| | | | | | | | |
|---|---|---|---|---|---|---|---|
| `all` | `and` | `any` | `ascii` | `at` | `base64` | `base64wide` | `condition` |
| `contains` | `endswith` | `entrypoint` | `false` | `filesize` | `for` | `fullword` | `global` |
| `import` | `icontains` | `iendswith` | `iequals` | `in` | `include` | `int16` | `int16be` |
| `int32` | `int32be` | `int8` | `int8be` | `istartswith` | `matches` | `meta` | `nocase` |
| `none` | `not` | `of` | `or` | `private` | `rule` | `startswith` | `strings` |
| `them` | `true` | `uint16` | `uint16be` | `uint32` | `uint32be` | `uint8` | `uint8be` |
| `wide` | `xor` | `defined` | | | | | |

Правила обычно состоят из двух секций: определение строк и условие. Секция определения строк может быть опущена, если правило не зависит от каких-либо строк, но секция условия обязательна всегда. Секция определения строк -- это место, где определяются строки, которые будут частью правила. Каждая строка имеет идентификатор, состоящий из символа `$`, за которым следует последовательность буквенно-цифровых символов и символов подчёркивания; эти идентификаторы могут использоваться в секции условия для ссылки на соответствующую строку. Строки могут быть определены в текстовом или шестнадцатеричном виде, как показано в следующем примере:

```yara
rule ExampleRule
{
    strings:
        $my_text_string = "text here"
        $my_hex_string = { E2 34 A1 C8 23 FB }

    condition:
        $my_text_string or $my_hex_string
}
```

Текстовые строки заключаются в двойные кавычки, как в языке C. Шестнадцатеричные строки заключаются в фигурные скобки и состоят из последовательности шестнадцатеричных чисел, которые могут следовать друг за другом без пробелов или быть разделены пробелами. Десятичные числа не допускаются в шестнадцатеричных строках.

Секция условия -- это место, где находится логика правила. Эта секция должна содержать логическое выражение, определяющее, при каких обстоятельствах файл или процесс удовлетворяет правилу. Как правило, условие ссылается на ранее определённые строки с помощью их идентификаторов. В этом контексте идентификатор строки действует как логическая переменная, которая принимает значение true, если строка найдена в файле или памяти процесса, и false в противном случае.

## Комментарии

Вы можете добавлять комментарии к вашим правилам YARA так же, как если бы это был исходный файл на C -- поддерживаются как однострочные, так и многострочные комментарии в стиле C.

```yara
/*
    Это многострочный комментарий ...
*/

rule CommentExample   // ... а это однострочный комментарий
{
    condition:
        false  // just a dummy rule, don't do this
}
```

## Строки

В YARA существует три типа строк: шестнадцатеричные строки, текстовые строки и регулярные выражения. Шестнадцатеричные строки используются для определения необработанных последовательностей байтов, в то время как текстовые строки и регулярные выражения полезны для определения фрагментов читаемого текста. Однако текстовые строки и регулярные выражения также могут использоваться для представления необработанных байтов с помощью escape-последовательностей, как будет показано ниже.

### Шестнадцатеричные строки

Шестнадцатеричные строки допускают четыре специальные конструкции, которые делают их более гибкими: подстановочные символы (wild-cards), операторы отрицания (not), переходы (jumps) и альтернативы (alternatives). Подстановочные символы -- это просто заполнители, которые можно поместить в строку, указывая, что некоторые байты неизвестны и должны совпадать с чем угодно. Символ-заполнитель -- это знак вопроса (?). Вот пример шестнадцатеричной строки с подстановочными символами:

```yara
rule WildcardExample
{
    strings:
        $hex_string = { E2 34 ?? C8 A? FB }

    condition:
        $hex_string
}
```

Как показано в примере, подстановочные символы работают на уровне полубайтов (nibble), что означает, что вы можете определить только один полубайт байта, а другой оставить неизвестным.

Начиная с версии 4.3.0, вы можете указать, что байт не является определённым значением. Для этого можно использовать оператор отрицания (not) с значением байта:

```yara
rule NotExample
{
    strings:
        $hex_string = { F4 23 ~00 62 B4 }
        $hex_string2 = { F4 23 ~?0 62 B4 }
    condition:
        $hex_string and $hex_string2
}
```

В приведённом выше примере у нас есть байт с префиксом тильда (~), который является оператором отрицания. Он определяет, что байт в этой позиции может принимать любое значение, кроме указанного. В данном случае первая строка совпадёт, только если байт не равен 00. Оператор отрицания также может использоваться с подстановочными символами на уровне полубайтов, поэтому вторая строка совпадёт, только если второй полубайт не равен нулю.

Подстановочные символы и операторы отрицания полезны при определении строк, содержимое которых может варьироваться, но длина переменных фрагментов известна. Однако это не всегда так. В некоторых случаях вам может потребоваться определить строки с фрагментами переменного содержимого и длины. В таких ситуациях вместо подстановочных символов можно использовать переходы (jumps):

```yara
rule JumpExample
{
    strings:
        $hex_string = { F4 23 [4-6] 62 B4 }

    condition:
        $hex_string
}
```

В приведённом выше примере мы видим пару чисел, заключённых в квадратные скобки и разделённых дефисом -- это переход. Этот переход указывает, что любая произвольная последовательность от 4 до 6 байтов может занимать позицию перехода. Любая из следующих строк будет соответствовать шаблону:

```
F4 23 01 02 03 04 62 B4
F4 23 00 00 00 00 00 62 B4
F4 23 15 82 A3 04 45 22 62 B4
```

Любой переход [X-Y] должен удовлетворять условию 0 <= X <= Y. В предыдущих версиях YARA оба значения X и Y должны были быть меньше 256, но начиная с YARA 2.0 ограничений на X и Y нет.

Допустимые переходы:

```
FE 39 45 [0-8] 89 00
FE 39 45 [23-45] 89 00
FE 39 45 [1000-2000] 89 00
```

Недопустимый переход:

```
FE 39 45 [10-7] 89 00
```

Если нижняя и верхняя границы равны, можно записать одно число в квадратных скобках, например:

```
FE 39 45 [6] 89 00
```

Приведённая выше строка эквивалентна обоим вариантам:

```
FE 39 45 [6-6] 89 00
FE 39 45 ?? ?? ?? ?? ?? ?? 89 00
```

Начиная с YARA 2.0 вы также можете использовать неограниченные переходы:

```
FE 39 45 [10-] 89 00
FE 39 45 [-] 89 00
```

Первый означает `[10-infinite]`, второй означает `[0-infinite]`.

Бывают также ситуации, когда вы хотите предоставить различные альтернативы для заданного фрагмента шестнадцатеричной строки. В таких случаях можно использовать синтаксис, напоминающий регулярные выражения:

```yara
rule AlternativesExample1
{
    strings:
        $hex_string = { F4 23 ( 62 B4 | 56 ) 45 }

    condition:
        $hex_string
}
```

Это правило сработает для любого файла, содержащего `F42362B445` или `F4235645`.

Можно указать и более двух альтернатив. Фактически нет ограничений ни на количество альтернативных последовательностей, ни на их длину.

```yara
rule AlternativesExample2
{
    strings:
        $hex_string = { F4 23 ( 62 B4 | 56 | 45 ?? 67 ) 45 }

    condition:
        $hex_string
}
```

Как видно из приведённого выше примера, строки, содержащие подстановочные символы, также допускаются в качестве альтернативных последовательностей.

### Текстовые строки

Как было показано в предыдущих разделах, текстовые строки обычно определяются следующим образом:

```yara
rule TextExample
{
    strings:
        $text_string = "foobar"

    condition:
        $text_string
}
```

Это простейший случай: строка в кодировке ASCII, чувствительная к регистру. Однако текстовые строки могут сопровождаться полезными модификаторами, которые изменяют способ интерпретации строки. Эти модификаторы добавляются в конце определения строки через пробел, как будет рассмотрено ниже.

Текстовые строки также могут содержать следующее подмножество escape-последовательностей, доступных в языке C:

| Последовательность | Описание |
|---|---|
| `\"` | Двойная кавычка |
| `\\` | Обратная косая черта |
| `\r` | Возврат каретки |
| `\t` | Горизонтальная табуляция |
| `\n` | Новая строка |
| `\xdd` | Любой байт в шестнадцатеричной нотации |

Во всех версиях YARA до 4.1.0 текстовые строки принимали любые символы Unicode независимо от их кодировки. Эти символы интерпретировались YARA как необработанные байты, и поэтому итоговая строка фактически определялась форматом кодировки, используемым вашим текстовым редактором. Это никогда не планировалось как функциональность -- изначально предполагалось, что строки YARA должны содержать только ASCII-символы. Начиная с YARA 4.1.0 стали выдаваться предупреждения о не-ASCII символах в строках. Это ограничение не распространяется на строки в секции метаданных и комментарии. Подробнее [здесь](https://github.com/VirusTotal/yara/wiki/Unicode-characters-in-YARA).

#### Строки без учёта регистра

Текстовые строки в YARA по умолчанию чувствительны к регистру, однако вы можете сделать строку нечувствительной к регистру, добавив модификатор `nocase` в конце определения строки, в той же строке:

```yara
rule CaseInsensitiveTextExample
{
    strings:
        $text_string = "foobar" nocase

    condition:
        $text_string
}
```

С модификатором `nocase` строка *foobar* будет совпадать с *Foobar*, *FOOBAR* и *fOoBaR*. Этот модификатор может использоваться в сочетании с любым другим модификатором, за исключением `base64`, `base64wide` и `xor`.

#### Строки с широкими символами

Модификатор `wide` может использоваться для поиска строк, закодированных двумя байтами на символ, что типично для многих исполняемых файлов.

Например, если строка "Borland" закодирована двумя байтами на символ (то есть `B\x00o\x00r\x00l\x00a\x00n\x00d\x00`), то следующее правило сработает:

```yara
rule WideCharTextExample1
{
    strings:
        $wide_string = "Borland" wide

    condition:
        $wide_string
}
```

Однако имейте в виду, что этот модификатор просто чередует ASCII-коды символов строки с нулями и не поддерживает истинные строки UTF-16, содержащие неанглийские символы. Если вы хотите искать строки как в ASCII, так и в широком формате, вы можете использовать модификатор `ascii` в сочетании с `wide`, независимо от порядка их следования.

```yara
rule WideCharTextExample2
{
    strings:
        $wide_and_ascii_string = "Borland" wide ascii

    condition:
        $wide_and_ascii_string
}
```

Модификатор `ascii` может использоваться отдельно, без сопутствующего модификатора `wide`, но писать его необязательно, поскольку в отсутствие `wide` строка по умолчанию считается ASCII.

#### Строки XOR

Модификатор `xor` может использоваться для поиска строк, к которым применена однобайтовая операция XOR.

Следующее правило будет искать все однобайтовые варианты XOR строки "This program cannot" (включая исходную строку в открытом виде):

```yara
rule XorExample1
{
    strings:
        $xor_string = "This program cannot" xor

    condition:
        $xor_string
}
```

Приведённое выше правило логически эквивалентно:

```yara
rule XorExample2
{
    strings:
        $xor_string_00 = "This program cannot"
        $xor_string_01 = "Uihr!qsnfs`l!b`oonu"
        $xor_string_02 = "Vjkq\"rpmepco\"acllmv"
        // Повторить для каждого однобайтового XOR
    condition:
        any of them
}
```

Вы также можете комбинировать модификатор `xor` с модификаторами `fullword`, `wide` и `ascii`. Например, чтобы искать версии строки в формате `wide` и `ascii` после применения каждого однобайтового XOR, используйте:

```yara
rule XorExample3
{
    strings:
        $xor_string = "This program cannot" xor wide ascii
    condition:
        $xor_string
}
```

Модификатор `xor` применяется после всех остальных модификаторов. Это означает, что совместное использование `xor` и `wide` приводит к тому, что XOR применяется к чередующимся нулевым байтам. Например, следующие два правила логически эквивалентны:

```yara
rule XorExample4
{
    strings:
        $xor_string = "This program cannot" xor wide
    condition:
        $xor_string
}

rule XorExample4
{
    strings:
        $xor_string_00 = "T\x00h\x00i\x00s\x00 \x00p\x00r\x00o\x00g\x00r\x00a\x00m\x00 \x00c\x00a\x00n\x00n\x00o\x00t\x00"
        $xor_string_01 = "U\x01i\x01h\x01r\x01!\x01q\x01s\x01n\x01f\x01s\x01`\x01l\x01!\x01b\x01`\x01o\x01o\x01n\x01u\x01"
        $xor_string_02 = "V\x02j\x02k\x02q\x02\"\x02r\x02p\x02m\x02e\x02p\x02c\x02o\x02\"\x02a\x02c\x02l\x02l\x02m\x02v\x02"
        // Повторить для каждой однобайтовой операции XOR.
    condition:
        any of them
}
```

Начиная с YARA 3.11, если вам нужен более точный контроль над диапазоном байтов, используемых с модификатором `xor`, используйте:

```yara
rule XorExample5
{
    strings:
        $xor_string = "This program cannot" xor(0x01-0xff)
    condition:
        $xor_string
}
```

Приведённый выше пример будет применять байты от 0x01 до 0xff включительно к строке при поиске. Общий синтаксис: `xor(минимум-максимум)`.

#### Строки Base64

Модификатор `base64` может использоваться для поиска строк, закодированных в base64. Хорошее объяснение этой техники можно найти по адресу:

https://www.leeholmes.com/searching-for-content-in-base-64-strings/

Следующее правило будет искать три перестановки base64-кодирования строки "This program cannot":

```yara
rule Base64Example1
{
    strings:
        $a = "This program cannot" base64

    condition:
        $a
}
```

Это заставит YARA искать следующие три перестановки:

| VGhpcyBwcm9ncmFtIGNhbm5vd |
|---|
| RoaXMgcHJvZ3JhbSBjYW5ub3 |
| UaGlzIHByb2dyYW0gY2Fubm90 |

Модификатор `base64wide` работает так же, как модификатор `base64`, но результаты модификатора `base64` преобразуются в широкий формат.

Взаимодействие между `base64` (или `base64wide`) и `wide` и `ascii` работает так, как можно ожидать. Сначала к строке применяются `wide` и `ascii`, а затем -- модификаторы `base64` и `base64wide`. При этом открытый текст ASCII- или широких версий строк не включается в поиск. Если вы хотите также включить их, вы можете поместить их во вторичную строку.

Модификаторы `base64` и `base64wide` также поддерживают пользовательский алфавит. Например:

```yara
rule Base64Example2
{
    strings:
        $a = "This program cannot" base64("!@#$%^&*(){}[].,|ABCDEFGHIJ\x09LMNOPQRSTUVWXYZabcdefghijklmnopqrstu")

    condition:
        $a
}
```

Алфавит должен быть длиной 64 байта.

Модификаторы `base64` и `base64wide` поддерживаются только с текстовыми строками. Использование этих модификаторов с шестнадцатеричной строкой или регулярным выражением вызовет ошибку компилятора. Также модификаторы `xor`, `fullword` и `nocase` в сочетании с `base64` или `base64wide` вызовут ошибку компилятора.

Из-за способа, которым YARA обрезает начальные и конечные символы после кодирования в base64, одно из base64-представлений "Dhis program cannow" и "This program cannot" идентично. Аналогично, использование ключевого слова `base64` с одиночными ASCII-символами не рекомендуется. Например, "a" с ключевым словом `base64` совпадает с "\`", "b", "c", "!", "\\xA1" или "\\xE1" после base64-кодирования и не будет совпадать там, где base64-кодирование соответствует регулярному выражению `[GWm2][EFGH]`.

#### Поиск полных слов

Ещё один модификатор, который может применяться к текстовым строкам, -- `fullword`. Этот модификатор гарантирует, что строка будет совпадать только в том случае, если она встречается в файле, ограниченная небуквенно-цифровыми символами. Например, строка *domain*, определённая как `fullword`, не совпадёт с *www.mydomain.com*, но совпадёт с *www.my-domain.com* и *www.domain.com*.

### Регулярные выражения

Регулярные выражения -- одна из самых мощных возможностей YARA. Они определяются так же, как текстовые строки, но заключаются в прямые косые черты вместо двойных кавычек, как в языке программирования Perl.

```yara
rule RegExpExample1
{
    strings:
        $re1 = /md5: [0-9a-fA-F]{32}/
        $re2 = /state: (on|off)/

    condition:
        $re1 and $re2
}
```

К регулярным выражениям также могут применяться модификаторы `nocase`, `ascii`, `wide` и `fullword`, как и к текстовым строкам. Семантика этих модификаторов одинакова в обоих случаях.

Кроме того, после закрывающей косой черты могут следовать символы `i` и `s` -- это очень распространённое соглашение для указания того, что регулярное выражение нечувствительно к регистру и что точка (`.`) может совпадать с символами новой строки. Например:

```yara
rule RegExpExample2
{
    strings:
        $re1 = /foo/i    // This regexp is case-insentitive
        $re2 = /bar./s   // In this regexp the dot matches everything, including new-line
        $re3 = /baz./is  // Both modifiers can be used together
    condition:
        any of them
}
```

Обратите внимание, что `/foo/i` эквивалентно `/foo/ nocase`, но мы рекомендуем последний вариант при определении строк. Синтаксис `/foo/i` полезен при написании нечувствительных к регистру регулярных выражений для оператора `matches`.

В предыдущих версиях YARA для сопоставления регулярных выражений использовались внешние библиотеки, такие как PCRE и RE2, но начиная с версии 2.0 YARA использует собственный движок регулярных выражений. Этот новый движок реализует большинство возможностей PCRE, за исключением нескольких: группы захвата, POSIX-классы символов ([[:isalpha:]], [[:isdigit:]] и т.д.) и обратные ссылки.

Регулярные выражения YARA распознают следующие метасимволы:

| Символ | Описание |
|---|---|
| `\` | Экранирование следующего метасимвола |
| `^` | Совпадение с началом файла или отрицание класса символов, если используется как первый символ после открывающей скобки |
| `$` | Совпадение с концом файла |
| `.` | Совпадение с любым одиночным символом, кроме символа новой строки |
| `\|` | Альтернатива |
| `()` | Группировка |
| `[]` | Класс символов в скобках |

Также распознаются следующие квантификаторы:

| Квантификатор | Описание |
|---|---|
| `*` | Совпадение 0 или более раз |
| `+` | Совпадение 1 или более раз |
| `?` | Совпадение 0 или 1 раз |
| `{n}` | Совпадение ровно n раз |
| `{n,}` | Совпадение не менее n раз |
| `{,m}` | Совпадение не более m раз |
| `{n,m}` | Совпадение от n до m раз |

Все эти квантификаторы имеют нежадный вариант, за которым следует знак вопроса (?):

| Квантификатор | Описание |
|---|---|
| `*?` | Совпадение 0 или более раз, нежадное |
| `+?` | Совпадение 1 или более раз, нежадное |
| `??` | Совпадение 0 или 1 раз, нежадное |
| `{n}?` | Совпадение ровно n раз, нежадное |
| `{n,}?` | Совпадение не менее n раз, нежадное |
| `{,m}?` | Совпадение не более m раз, нежадное |
| `{n,m}?` | Совпадение от n до m раз, нежадное |

Распознаются следующие escape-последовательности:

| Последовательность | Описание |
|---|---|
| `\t` | Табуляция (HT, TAB) |
| `\n` | Новая строка (LF, NL) |
| `\r` | Возврат каретки (CR) |
| `\f` | Перевод страницы (FF) |
| `\a` | Звуковой сигнал |
| `\xNN` | Символ, порядковый номер которого задан шестнадцатеричным числом |

Распознаваемые классы символов:

| Класс | Описание |
|---|---|
| `\w` | Совпадение с символом *слова* (буквенно-цифровой символ плюс "_") |
| `\W` | Совпадение с символом, *не являющимся словом* |
| `\s` | Совпадение с пробельным символом |
| `\S` | Совпадение с непробельным символом |
| `\d` | Совпадение с десятичной цифрой |
| `\D` | Совпадение с нецифровым символом |

Начиная с версии 3.3.0 также распознаются следующие утверждения нулевой ширины:

| Утверждение | Описание |
|---|---|
| `\b` | Совпадение с границей слова |
| `\B` | Совпадение всюду, кроме границы слова |

### Приватные строки

Все строки в YARA могут быть помечены как `private`, что означает, что они никогда не будут включены в вывод YARA. В остальном они обрабатываются как обычные строки, поэтому вы можете использовать их в условии по своему усмотрению, но они никогда не будут показаны с флагом `-s` и не будут видны в обратном вызове YARA при использовании C API.

```yara
rule PrivateStringExample
{
    strings:
        $text_string = "foobar" private

    condition:
        $text_string
}
```

### Неиспользуемые строки

YARA 4.5.0 допускает неиспользуемые строки в условии. Если идентификатор строки начинается с `_`, он не обязан упоминаться в условии. Любая другая строка должна быть упомянута в условии. Это полезно, если вы хотите искать определённые строки и обрабатывать их в пользовательском обратном вызове, но они не нужны для логики условия.

```yara
rule PrivateStringExample
{
    strings:
        $_unreferenced = "AXSERS"

    condition:
        true
}
```

### Сводка модификаторов строк

Следующие модификаторы строк обрабатываются в указанном порядке, но применимы только к перечисленным типам строк.

| Ключевое слово | Типы строк | Описание | Ограничения |
|---|---|---|---|
| `nocase` | Текстовые, Регулярные выражения | Игнорировать регистр | Нельзя использовать с `xor`, `base64` или `base64wide` |
| `wide` | Текстовые, Регулярные выражения | Эмулировать UTF16 чередованием нулевых символов (0x00) | Нет |
| `ascii` | Текстовые, Регулярные выражения | Также совпадать с ASCII-символами, требуется только если используется `wide` | Нет |
| `xor` | Текстовые | XOR текстовой строки с однобайтовыми ключами | Нельзя использовать с `nocase`, `base64` или `base64wide` |
| `base64` | Текстовые | Преобразовать в 3 строки, закодированные в base64 | Нельзя использовать с `nocase`, `xor` или `fullword` |
| `base64wide` | Текстовые | Преобразовать в 3 строки base64, затем чередовать нулевые символы как `wide` | Нельзя использовать с `nocase`, `xor` или `fullword` |
| `fullword` | Текстовые, Регулярные выражения | Совпадению не предшествует и не следует буквенно-цифровой символ | Нельзя использовать с `base64` или `base64wide` |
| `private` | Шестнадцатеричные, Текстовые, Регулярные выражения | Совпадение никогда не включается в вывод | Нет |

## Условия

Условия -- это не что иное, как логические выражения, подобные тем, которые встречаются во всех языках программирования, например, в операторе *if*. Они могут содержать типичные логические операторы `and`, `or` и `not`, а также операторы сравнения `>=`, `<=`, `<`, `>`, `==` и `!=`. Также в числовых выражениях могут использоваться арифметические операторы (`+`, `-`, `*`, `\`, `%`) и побитовые операторы (`&`, `|`, `<<`, `>>`, `~`, `^`).

Целые числа всегда 64-битные, даже результаты таких функций, как `uint8`, `uint16` и `uint32`, приводятся к 64 битам. Это необходимо учитывать, особенно при использовании побитовых операторов (например, ~0x01 -- это не 0xFE, а 0xFFFFFFFFFFFFFFFE).

В следующей таблице перечислены приоритет и ассоциативность всех операторов. Таблица отсортирована в порядке убывания приоритета, что означает, что операторы в верхних строках группируются раньше операторов в нижних строках. Операторы в одной строке имеют одинаковый приоритет, и если они встречаются вместе в выражении, их группировка определяется ассоциативностью.

| Приоритет | Оператор | Описание | Ассоциативность |
|---|---|---|---|
| 1 | `[]` | Индексация массива | Слева направо |
|   | `.` | Доступ к полю структуры | |
| 2 | `-` | Унарный минус | Справа налево |
|   | `~` | Побитовое НЕ | |
| 3 | `*` | Умножение | Слева направо |
|   | `\` | Деление | |
|   | `%` | Остаток от деления | |
| 4 | `+` | Сложение | Слева направо |
|   | `-` | Вычитание | |
| 5 | `<<` | Побитовый сдвиг влево | Слева направо |
|   | `>>` | Побитовый сдвиг вправо | |
| 6 | `&` | Побитовое И | Слева направо |
| 7 | `^` | Побитовое исключающее ИЛИ | Слева направо |
| 8 | `\|` | Побитовое ИЛИ | Слева направо |
| 9 | `<` | Меньше | Слева направо |
|   | `<=` | Меньше или равно | |
|   | `>` | Больше | |
|   | `>=` | Больше или равно | |
| 10 | `==` | Равно | Слева направо |
|   | `!=` | Не равно | |
|   | `contains` | Строка содержит подстроку | |
|   | `icontains` | Как contains, но без учёта регистра | |
|   | `startswith` | Строка начинается с подстроки | |
|   | `istartswith` | Как startswith, но без учёта регистра | |
|   | `endswith` | Строка заканчивается подстрокой | |
|   | `iendswith` | Как endswith, но без учёта регистра | |
|   | `iequals` | Сравнение строк без учёта регистра | |
|   | `matches` | Строка соответствует регулярному выражению | |
| 11 | `not` | Логическое НЕ | Справа налево |
|   | `defined` | Проверка, определено ли выражение | |
| 12 | `and` | Логическое И | Слева направо |
| 13 | `or` | Логическое ИЛИ | Слева направо |

Идентификаторы строк также могут использоваться в условии, действуя как логические переменные, значение которых зависит от наличия или отсутствия соответствующей строки в файле.

```yara
rule Example
{
    strings:
        $a = "text1"
        $b = "text2"
        $c = "text3"
        $d = "text4"

    condition:
        ($a or $b) and ($c or $d)
}
```

### Подсчёт строк

Иногда нам нужно знать не только, присутствует ли определённая строка, но и сколько раз она встречается в файле или памяти процесса. Количество вхождений каждой строки представлено переменной, имя которой совпадает с идентификатором строки, но с символом `#` вместо символа `$`. Например:

```yara
rule CountExample
{
    strings:
        $a = "dummy1"
        $b = "dummy2"

    condition:
        #a == 6 and #b > 10
}
```

Это правило сработает для любого файла или процесса, содержащего строку $a ровно шесть раз и более десяти вхождений строки $b.

Начиная с YARA 4.2.0 можно выразить подсчёт строки в целочисленном диапазоне, например:

```yara
#a in (filesize-500..filesize) == 2
```

В этом примере количество строк 'a' в последних 500 байтах файла должно быть ровно 2.

### Смещения строк или виртуальные адреса

В большинстве случаев, когда идентификатор строки используется в условии, мы хотим узнать, находится ли соответствующая строка где-либо в файле или памяти процесса. Но иногда нам нужно знать, находится ли строка по определённому смещению в файле или по определённому виртуальному адресу в адресном пространстве процесса. В таких ситуациях нужен оператор `at`. Этот оператор используется, как показано в следующем примере:

```yara
rule AtExample
{
    strings:
        $a = "dummy1"
        $b = "dummy2"

    condition:
        $a at 100 and $b at 200
}
```

Выражение `$a at 100` в приведённом выше примере истинно, только если строка $a найдена по смещению 100 в файле (или по виртуальному адресу 100, если применяется к запущенному процессу). Строка $b должна быть по смещению 200. Обратите внимание, что оба смещения десятичные, однако шестнадцатеричные числа можно записать, добавив префикс 0x перед числом, как в языке C, что очень удобно при записи виртуальных адресов. Также обратите внимание на более высокий приоритет оператора `at` по сравнению с `and`.

В то время как оператор `at` позволяет искать строку по фиксированному смещению в файле или виртуальному адресу в адресном пространстве процесса, оператор `in` позволяет искать строку в диапазоне смещений или адресов.

```yara
rule InExample
{
    strings:
        $a = "dummy1"
        $b = "dummy2"

    condition:
        $a in (0..100) and $b in (100..filesize)
}
```

В приведённом примере строка $a должна быть найдена по смещению от 0 до 100, а строка $b -- по смещению от 100 до конца файла. Опять же, числа по умолчанию десятичные.

Вы также можете получить смещение или виртуальный адрес i-го вхождения строки $a, используя `@a[i]`. Индексы начинаются с единицы, поэтому первое вхождение будет `@a[1]`, второе -- `@a[2]` и так далее. Если вы укажете индекс, превышающий количество вхождений строки, результатом будет NaN (Not A Number -- не число).

### Длина совпадения

Для многих регулярных выражений и шестнадцатеричных строк, содержащих переходы, длина совпадения является переменной. Если у вас есть регулярное выражение /fo*/, то совпадениями могут быть строки "fo", "foo" и "fooo", каждая с разной длиной.

Вы можете использовать длину совпадений как часть условия, используя символ `!` перед идентификатором строки, аналогично тому, как вы используете символ `@` для смещения. `!a[1]` -- это длина первого совпадения $a, `!a[2]` -- длина второго совпадения и так далее. `!a` -- это сокращённая форма `!a[1]`.

### Размер файла

Идентификаторы строк -- не единственные переменные, которые могут использоваться в условии (на самом деле, правила могут быть определены вообще без определения строк, как будет показано ниже), существуют и другие специальные переменные. Одна из них -- `filesize`, которая содержит, как следует из названия, размер сканируемого файла. Размер выражается в байтах.

```yara
rule FileSizeExample
{
    condition:
        filesize > 200KB
}
```

Предыдущий пример также демонстрирует использование постфикса `KB`. Этот постфикс, присоединённый к числовой константе, автоматически умножает значение константы на 1024. Постфикс `MB` может использоваться для умножения значения на 2^20. Оба постфикса могут использоваться только с десятичными константами.

Использование `filesize` имеет смысл только при применении правила к файлу. Если правило применяется к запущенному процессу, оно никогда не сработает, поскольку `filesize` не имеет смысла в этом контексте.

### Точка входа исполняемого файла

Ещё одна специальная переменная, которая может использоваться в правиле, -- `entrypoint`. Если файл является Portable Executable (PE) или Executable and Linkable Format (ELF), эта переменная содержит необработанное смещение точки входа исполняемого файла при сканировании файла. При сканировании запущенного процесса entrypoint содержит виртуальный адрес точки входа основного исполняемого файла. Типичное использование этой переменной -- поиск определённого шаблона в точке входа для обнаружения упаковщиков или простых файловых инфекторов.

```yara
rule EntryPointExample1
{
    strings:
        $a = { E8 00 00 00 00 }

    condition:
        $a at entrypoint
}

rule EntryPointExample2
{
    strings:
        $a = { 9C 50 66 A1 ?? ?? ?? 00 66 A9 ?? ?? 58 0F 85 }

    condition:
        $a in (entrypoint..entrypoint + 10)
}
```

Наличие переменной `entrypoint` в правиле подразумевает, что только файлы PE или ELF могут удовлетворить это правило. Если файл не является PE или ELF, любое правило, использующее эту переменную, вычисляется как false.

> **Предупреждение:** Переменная `entrypoint` устарела, вместо неё следует использовать эквивалентную `pe.entry_point` из модуля PE. Начиная с YARA 3.0, при использовании `entrypoint` вы получите предупреждение, и в будущих версиях она будет полностью удалена.

### Доступ к данным по заданной позиции

Существует множество ситуаций, когда вы хотите написать условия, зависящие от данных, хранящихся по определённому смещению файла или виртуальному адресу памяти, в зависимости от того, сканируем ли мы файл или запущенный процесс. В таких ситуациях вы можете использовать одну из следующих функций для чтения данных из файла по заданному смещению:

```
int8(<смещение или виртуальный адрес>)
int16(<смещение или виртуальный адрес>)
int32(<смещение или виртуальный адрес>)

uint8(<смещение или виртуальный адрес>)
uint16(<смещение или виртуальный адрес>)
uint32(<смещение или виртуальный адрес>)

int8be(<смещение или виртуальный адрес>)
int16be(<смещение или виртуальный адрес>)
int32be(<смещение или виртуальный адрес>)

uint8be(<смещение или виртуальный адрес>)
uint16be(<смещение или виртуальный адрес>)
uint32be(<смещение или виртуальный адрес>)
```

Функции `intXX` читают 8-, 16- и 32-битные знаковые целые числа из <смещения или виртуального адреса>, а функции `uintXX` читают беззнаковые целые числа. Как 16-, так и 32-битные целые числа считаются записанными в формате little-endian. Если вы хотите прочитать целое число в формате big-endian, используйте соответствующую функцию с окончанием `be`. Параметр <смещение или виртуальный адрес> может быть любым выражением, возвращающим беззнаковое целое число, включая возвращаемое значение самих функций `uintXX`. В качестве примера рассмотрим правило для распознавания PE-файлов:

```yara
rule IsPE
{
    condition:
        // MZ signature at offset 0 and ...
        uint16(0) == 0x5A4D and
        // ... PE signature at offset stored in MZ header at 0x3C
        uint32(uint32(0x3C)) == 0x00004550
}
```

### Наборы строк

Бывают случаи, когда необходимо указать, что файл должен содержать определённое количество строк из заданного набора. Ни одна из строк набора не обязана присутствовать, но хотя бы некоторые из них должны быть. В таких ситуациях можно использовать оператор `of`.

```yara
rule OfExample1
{
    strings:
        $a = "dummy1"
        $b = "dummy2"
        $c = "dummy3"

    condition:
        2 of ($a,$b,$c)
}
```

Это правило требует, чтобы в файле присутствовали как минимум две строки из набора ($a,$b,$c), но неважно, какие именно. Разумеется, при использовании этого оператора число перед ключевым словом `of` должно быть меньше или равно количеству строк в наборе.

Элементы набора могут быть перечислены явно, как в предыдущем примере, или заданы с помощью подстановочных символов. Например:

```yara
rule OfExample2
{
    strings:
        $foo1 = "foo1"
        $foo2 = "foo2"
        $foo3 = "foo3"

    condition:
        2 of ($foo*)  // equivalent to 2 of ($foo1,$foo2,$foo3)
}

rule OfExample3
{
    strings:
        $foo1 = "foo1"
        $foo2 = "foo2"

        $bar1 = "bar1"
        $bar2 = "bar2"

    condition:
        3 of ($foo*,$bar1,$bar2)
}
```

Вы даже можете использовать `($*)` для ссылки на все строки в вашем правиле или написать эквивалентное ключевое слово `them` для лучшей читаемости.

```yara
rule OfExample4
{
    strings:
        $a = "dummy1"
        $b = "dummy2"
        $c = "dummy3"

    condition:
        1 of them // equivalent to 1 of ($*)
}
```

Во всех приведённых выше примерах количество строк задавалось числовой константой, но можно использовать любое выражение, возвращающее числовое значение. Также могут использоваться ключевые слова `any`, `all` и `none`.

```yara
all of them       // all strings in the rule
any of them       // any string in the rule
all of ($a*)      // all strings whose identifier starts by $a
any of ($a,$b,$c) // any of $a, $b or $c
1 of ($*)         // same that "any of them"
none of ($b*)     // zero of the set of strings that start with "$b"
```

> **Предупреждение:** Из-за особенностей внутренней работы YARA использование "0 of them" является неоднозначной частью языка, которой следует избегать в пользу "none of them". Чтобы понять это, рассмотрим значение "2 of them", которое истинно, если 2 или более строк совпадают. Исторически "0 of them" следовало этому принципу и принимало значение true, если хотя бы одна из строк совпадала. Эта неоднозначность разрешена в YARA 4.3.0, где "0 of them" принимает значение true, если совпадает ровно 0 строк. Для улучшения ситуации и ясности намерения рекомендуется использовать "none" вместо 0. Не используя целое число, легче понять смысл "none of them" без исторического понимания "не менее 0", затуманивающего суть.

Начиная с YARA 4.2.0 можно выразить набор строк в целочисленном диапазоне, например:

```yara
all of ($a*) in (filesize-500..filesize)
any of ($a*, $b*) in (1000..2000)
```

Начиная с YARA 4.3.0 можно выразить набор строк по конкретному смещению, например:

```yara
any of ($a*) at 0
```

### Применение одного условия ко множеству строк

Существует ещё один оператор, очень похожий на `of`, но ещё более мощный -- оператор `for..of`. Его синтаксис:

```yara
for expression of string_set : ( boolean_expression )
```

Его значение: из строк в `string_set` как минимум `expression` из них должны удовлетворять `boolean_expression`.

Другими словами: `boolean_expression` вычисляется для каждой строки в `string_set`, и как минимум `expression` из них должны вернуть True.

Разумеется, `boolean_expression` может быть любым логическим выражением, допустимым в секции условия правила, за одним важным исключением: здесь вы можете (и должны) использовать знак доллара ($) в качестве заполнителя для текущей вычисляемой строки. Рассмотрим следующее выражение:

```yara
for any of ($a,$b,$c) : ( $ at pe.entry_point  )
```

Символ $ в логическом выражении не привязан к какой-либо конкретной строке -- он будет последовательно заменён на $a, затем $b, затем $c при каждом вычислении выражения.

Возможно, вы уже заметили, что оператор `of` является частным случаем `for..of`. Следующие выражения эквивалентны:

```yara
any of ($a,$b,$c)
for any of ($a,$b,$c) : ( $ )
```

Вы также можете использовать символы `#`, `@` и `!` для ссылки соответственно на количество вхождений, первое смещение и длину каждой строки.

```yara
for all of them : ( # > 3 )
for all of ($a*) : ( @ > @b )
```

Начиная с YARA 4.3.0 вы можете задавать условия над текстовыми строками следующим образом:

```yara
for any s in ("71b36345516e076a0663e0bea97759e4", "1e7f7edeb06de02f2c2a9319de99e033") : ( pe.imphash() == s )
```

Стоит отметить, что два хеша, упомянутые в правиле, являются обычными текстовыми строками и не имеют отношения к секции строк правила. Внутри условия цикла результат функции `pe.imphash()` сравнивается с каждой из текстовых строк, что делает правило более лаконичным.

### Использование анонимных строк с `of` и `for..of`

При использовании операторов `of` и `for..of` с `them` идентификатор, присвоенный каждой строке правила, обычно является избыточным. Поскольку мы не ссылаемся на какую-либо строку по отдельности, нет необходимости предоставлять уникальный идентификатор для каждой из них. В таких ситуациях можно объявить анонимные строки с идентификаторами, состоящими только из символа $, как в следующем примере:

```yara
rule AnonymousStrings
{
    strings:
        $ = "dummy1"
        $ = "dummy2"

    condition:
        1 of them
}
```

### Итерация по вхождениям строк

Как было показано в разделе [Смещения строк или виртуальные адреса](#смещения-строк-или-виртуальные-адреса), доступ к смещениям или виртуальным адресам, по которым данная строка появляется в файле или адресном пространстве процесса, осуществляется с помощью синтаксиса: `@a[i]`, где i -- индекс, указывающий, какое вхождение строки $a вы имеете в виду (@a[1], @a[2] и т.д.).

Иногда вам потребуется выполнить итерацию по некоторым из этих смещений и убедиться, что они удовлетворяют заданному условию. В таких случаях можно использовать синтаксис `for..in`, например:

```yara
rule Occurrences
{
    strings:
        $a = "dummy1"
        $b = "dummy2"

    condition:
        for all i in (1,2,3) : ( @a[i] + 10 == @b[i] )
}
```

Предыдущее правило гласит, что первое вхождение $b должно быть на 10 байтов позже первого вхождения $a, и то же самое должно выполняться для второго и третьего вхождений обеих строк.

То же условие можно записать и так:

```yara
for all i in (1..3) : ( @a[i] + 10 == @b[i] )
```

Обратите внимание, что мы используем диапазон (1..3) вместо перечисления значений индекса (1,2,3). Разумеется, мы не обязаны использовать константы для задания границ диапазона -- можно использовать выражения, как в следующем примере:

```yara
for all i in (1..#a) : ( @a[i] < 100 )
```

В данном случае мы выполняем итерацию по каждому вхождению $a (помните, что #a представляет количество вхождений $a). Это правило указывает, что каждое вхождение $a должно находиться в первых 100 байтах файла.

Если вы хотите указать, что только некоторые вхождения строки должны удовлетворять условию, применяется та же логика, что и в операторе `for..of`:

```yara
for any i in (1..#a) : ( @a[i] < 100 )
for 2 i in (1..#a) : ( @a[i] < 100 )
```

Оператор `for..in` похож на `for..of`, но последний выполняет итерацию по набору строк, тогда как первый -- по диапазонам, перечислениям, массивам и словарям.

### Итераторы

В YARA 4.0 оператор `for..in` был улучшен и теперь может использоваться для итерации не только по целочисленным перечислениям и диапазонам (например, 1,2,3,4 и 1..4), но и по любому итерируемому типу данных, такому как массивы и словари, определённые модулями YARA. Например, следующее выражение допустимо в YARA 4.0:

```yara
for any section in pe.sections : ( section.name == ".text" )
```

Это эквивалентно:

```yara
for any i in (0..pe.number_of_sections-1) : ( pe.sections[i].name == ".text" )
```

Новый синтаксис более естественный и понятный, и является рекомендуемым способом выражения подобных условий в новых версиях YARA.

При итерации по словарям необходимо указать два имени переменных, которые будут содержать ключ и значение для каждой записи словаря, например:

```yara
for any k,v in some_dict : ( k == "foo" and v == "bar" )
```

В общем виде оператор `for..in` имеет форму:

```yara
for <quantifier> <variables> in <iterable> : ( <некоторое условие с использованием переменных цикла> )
```

Где `<quantifier>` -- это `any`, `all` или выражение, вычисляемое в количество элементов итератора, которые должны удовлетворять условию; `<variables>` -- это разделённый запятыми список имён переменных, содержащих значения текущего элемента (количество переменных зависит от типа `<iterable>`); и `<iterable>` -- это нечто, по чему можно выполнить итерацию.

### Ссылки на другие правила

При написании условия правила вы также можете ссылаться на ранее определённое правило способом, напоминающим вызов функции в традиционных языках программирования. Таким образом можно создавать правила, зависящие от других. Рассмотрим пример:

```yara
rule Rule1
{
    strings:
        $a = "dummy1"

    condition:
        $a
}

rule Rule2
{
    strings:
        $a = "dummy2"

    condition:
        $a and Rule1
}
```

Как видно из примера, файл удовлетворит Rule2 только если он содержит строку "dummy2" и удовлетворяет Rule1. Обратите внимание, что правило, на которое ссылаются, обязательно должно быть определено до правила, которое содержит ссылку.

Другой способ ссылки на другие правила был введён в версии 4.2.0 -- наборы правил, которые работают аналогично наборам строк (см. [Наборы строк](#наборы-строк)). Например:

```yara
rule Rule1
{
    strings:
        $a = "dummy1"

    condition:
        $a
}

rule Rule2
{
    strings:
        $a = "dummy2"

    condition:
        $a
}

rule MainRule
{
    strings:
        $a = "dummy2"

    condition:
        any of (Rule*)
}
```

Этот пример демонстрирует, как использовать наборы правил для описания логики более высокого порядка, которая автоматически расширяется вместе с вашими правилами. Если вы определите другое правило с именем `Rule3` перед `MainRule`, оно автоматически будет включено в раскрытие `Rule*` в условии MainRule.

Для использования наборов правил все правила, включённые в набор, **должны** существовать до того, как набор правил будет использован. Например, следующий код вызовет ошибку компилятора, потому что `a2` определено после использования набора правил в `x`:

```yara
rule a1 { condition: true }
rule x { condition: 1 of (a*) }
rule a2 { condition: true }
```

## Дополнительные сведения о правилах

Есть некоторые аспекты правил YARA, которые ещё не были рассмотрены, но остаются очень важными. Это: глобальные правила, приватные правила, теги и метаданные.

### Глобальные правила

Глобальные правила позволяют одновременно накладывать ограничения на все ваши правила. Например, предположим, что вы хотите, чтобы все ваши правила игнорировали файлы, превышающие определённый размер. Вы могли бы пройти по каждому правилу и внести необходимые изменения в условия, или просто написать глобальное правило:

```yara
global rule SizeLimit
{
    condition:
        filesize < 2MB
}
```

Вы можете определить сколько угодно глобальных правил -- они будут вычислены перед остальными правилами, которые, в свою очередь, будут вычислены только при выполнении всех глобальных правил.

### Приватные правила

Приватные правила -- очень простая концепция. Это просто правила, о которых YARA не сообщает при их срабатывании на файле. Правила, которые вообще не показываются в отчётах, на первый взгляд могут показаться бесполезными, но в сочетании с возможностью ссылки одного правила на другое (см. [Ссылки на другие правила](#ссылки-на-другие-правила)) они становятся полезными. Приватные правила могут служить строительными блоками для других правил и одновременно предотвращать засорение вывода YARA нерелевантной информацией. Чтобы объявить правило как приватное, просто добавьте ключевое слово `private` перед объявлением правила.

```yara
private rule PrivateRuleExample
{
    ...
}
```

Вы можете применить к правилу оба модификатора -- `private` и `global`, получив глобальное правило, которое не отображается в выводе YARA, но должно быть выполнено.

### Теги правил

Ещё одна полезная возможность YARA -- добавление тегов к правилам. Эти теги могут использоваться впоследствии для фильтрации вывода YARA и отображения только тех правил, которые вас интересуют. Вы можете добавить к правилу сколько угодно тегов; они объявляются после идентификатора правила, как показано ниже:

```yara
rule TagsExample1 : Foo Bar Baz
{
    ...
}

rule TagsExample2 : Bar
{
    ...
}
```

Теги должны следовать тем же лексическим соглашениям, что и идентификаторы правил: допускаются только буквенно-цифровые символы и символ подчёркивания, и тег не может начинаться с цифры. Они также чувствительны к регистру.

При использовании YARA вы можете выводить только те правила, которые отмечены указанным вами тегом или тегами.

### Метаданные

Помимо секций определения строк и условий, правила могут также иметь секцию метаданных, в которой можно указать дополнительную информацию о правиле. Секция метаданных определяется ключевым словом `meta` и содержит пары идентификатор/значение, как в следующем примере:

```yara
rule MetadataExample
{
    meta:
        my_identifier_1 = "Some string data"
        my_identifier_2 = 24
        my_identifier_3 = true

    strings:
        $my_text_string = "text here"
        $my_hex_string = { E2 34 A1 C8 23 FB }

    condition:
        $my_text_string or $my_hex_string
}
```

Как видно из примера, за идентификаторами метаданных всегда следует знак равенства и присвоенное им значение. Присваиваемые значения могут быть строками (только допустимый UTF8), целыми числами или одним из логических значений true или false. Обратите внимание, что пары идентификатор/значение, определённые в секции метаданных, не могут использоваться в секции условия -- их единственная цель заключается в хранении дополнительной информации о правиле.

## Использование модулей

Модули -- это расширения основной функциональности YARA. Некоторые модули, такие как модуль PE и модуль Cuckoo, официально распространяются с YARA, а дополнительные могут быть созданы третьими сторонами или даже вами самостоятельно.

Первым шагом к использованию модуля является его импорт с помощью оператора `import`. Эти операторы должны располагаться вне определения любого правила и содержать имя модуля, заключённое в двойные кавычки. Например:

```yara
import "pe"
import "cuckoo"
```

После импорта модуля вы можете использовать его возможности, всегда указывая `<имя модуля>.` в качестве префикса к любой переменной или функции, экспортируемой модулем. Например:

```yara
pe.entry_point == 0x1000
cuckoo.http_request(/someregexp/)
```

## Неопределённые значения

Модули часто оставляют переменные в неопределённом состоянии, например, когда переменная не имеет смысла в текущем контексте (представьте `pe.entry_point` при сканировании файла, не являющегося PE). YARA обрабатывает неопределённые значения таким образом, чтобы правило сохраняло свой смысл. Рассмотрим это правило:

```yara
import "pe"

rule Test
{
    strings:
        $a = "some string"

    condition:
        $a and pe.entry_point == 0x1000
}
```

Если сканируемый файл не является PE, вы не ожидали бы, что это правило сработает, даже если файл содержит строку, потому что **оба** условия (наличие строки и правильное значение точки входа) должны быть выполнены. Однако, если условие изменить на:

```yara
$a or pe.entry_point == 0x1000
```

Вы ожидали бы, что правило сработает, если файл содержит строку, даже если это не PE-файл. Именно так YARA и работает. Логика следующая:

* Если выражение в условии не определено, оно преобразуется в `false`, и правило не сработает.

* Логические операторы `and` и `or` обрабатывают неопределённые операнды как `false`, что означает:

  * `undefined and true` = `false`
  * `undefined and false` = `false`
  * `undefined or true` = `true`
  * `undefined or false` = `false`

* Все остальные операторы, включая оператор `not`, возвращают undefined, если любой из их операндов не определён.

В приведённом выше выражении `pe.entry_point == 0x1000` будет undefined для не-PE файлов, потому что `pe.entry_point` не определён для таких файлов. Это означает, что `$a or pe.entry_point == 0x1000` будет `true` тогда и только тогда, когда `$a` истинно.

Если условие состоит только из `pe.entry_point == 0x1000`, оно будет вычислено как `false` для не-PE файлов, как и `pe.entry_point != 0x1000` и `not pe.entry_point == 0x1000`, поскольку ни одно из этих выражений не имеет смысла для не-PE файлов.

Чтобы проверить, определено ли выражение, используйте унарный оператор `defined`. Пример:

```yara
defined pe.entry_point
```

## Внешние переменные

Внешние переменные позволяют определять правила, зависящие от значений, предоставляемых извне. Например, вы можете написать следующее правило:

```yara
rule ExternalVariableExample1
{
    condition:
        ext_var == 10
}
```

В данном случае `ext_var` -- это внешняя переменная, значение которой присваивается во время выполнения (см. параметр `-d` инструмента командной строки и параметр `externals` методов `compile` и `match` в yara-python). Внешние переменные могут быть типов: целое число, строка или логическое значение; их тип определяется присвоенным им значением. Целочисленная переменная может заменить любую целочисленную константу в условии, а логические переменные могут занять место логических выражений. Например:

```yara
rule ExternalVariableExample2
{
    condition:
        bool_ext_var or filesize < int_ext_var
}
```

Внешние переменные строкового типа могут использоваться с операторами: `contains`, `startswith`, `endswith` и их регистронезависимыми аналогами: `icontains`, `istartswith` и `iendswith`. Они также могут использоваться с оператором `matches`, который возвращает true, если строка соответствует заданному регулярному выражению. Регистронезависимое сравнение строк выполняется с помощью специального оператора `iequals`, который работает только со строками. Для регистрозависимого сравнения используйте обычный `==`.

```yara
rule ContainsExample
{
    condition:
        string_ext_var contains "text"
}

rule CaseInsensitiveContainsExample
{
    condition:
        string_ext_var icontains "text"
}

rule StartsWithExample
{
    condition:
        string_ext_var startswith "prefix"
}

rule EndsWithExample
{
    condition:
        string_ext_var endswith "suffix"
}

rule IequalsExample
{
    condition:
        string_ext_var iequals "string"
}

rule MatchesExample
{
    condition:
        string_ext_var matches /[a-z]+/
}
```

Вы можете использовать модификаторы регулярных выражений вместе с оператором `matches`. Например, если вы хотите сделать регулярное выражение из предыдущего примера нечувствительным к регистру, вы можете использовать `/[a-z]+/i`. Обратите внимание на `i` после регулярного выражения в стиле Perl. Вы также можете использовать модификатор `s` для однострочного режима, в котором точка совпадает со всеми символами, включая переносы строк. Разумеется, оба модификатора могут использоваться одновременно, как в следующем примере:

```yara
rule ExternalVariableExample5
{
    condition:
        /* case insensitive single-line mode */
        string_ext_var matches /[a-z]+/is
}
```

Имейте в виду, что каждая внешняя переменная, используемая в ваших правилах, должна быть определена во время выполнения -- либо с помощью параметра `-d` инструмента командной строки, либо путём указания параметра `externals` соответствующему методу в `yara-python`.

## Включение файлов

Для более гибкой организации файлов правил YARA предоставляет директиву `include`. Эта директива работает аналогично директиве препроцессора *#include* в программах на C, которая вставляет содержимое указанного исходного файла в текущий файл при компиляции. Следующий пример включит содержимое *other.yar* в текущий файл:

```yara
include "other.yar"
```

Базовый путь при поиске файла в директиве `include` -- это каталог, в котором находится текущий файл. По этой причине файл *other.yar* в предыдущем примере должен находиться в том же каталоге, что и текущий файл. Однако вы также можете указать относительные пути:

```yara
include "./includes/other.yar"
include "../includes/other.yar"
```

Или использовать абсолютные пути:

```yara
include "/home/plusvic/yara/includes/other.yar"
```

В Windows допускаются как прямые, так и обратные косые черты, но не забудьте указать букву диска:

```yara
include "c:/yara/includes/other.yar"
include "c:\\yara\\includes\\other.yar"
```
